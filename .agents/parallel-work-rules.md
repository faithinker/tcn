# TCN 프로젝트 병렬 작업 규칙 (Parallel Working Rules)

이 문서는 개발자(인간)와 여러 AI 에이전트가 동일한 저장소에서 동시/병렬적으로 안전하게 작업을 수행하고 브랜치 및 파일 충돌을 방지하기 위한 협업 표준 규칙입니다.

---

## 1. Git Worktree를 통한 물리적 공간 격리
* **동일 디렉토리 내 브랜치 전환 제한:**
  * 동일한 작업 디렉토리 내에서 `git checkout`을 무분별하게 변경하면, 다른 병렬 프로세스(또는 다른 에이전트)의 컴파일/빌드 환경이 깨질 수 있습니다.
* **해결책:** 
  * 에이전트 소환 시 `Workspace` 모드를 `share`(git worktree 공유) 또는 `branch`(격리 복제)로 설정하여 독립된 폴더 환경에서 개발을 진행합니다.
  * 로컬에서 두 개 이상의 기능을 동시 작업해야 하는 경우 아래 명령어를 사용하여 독립 폴더를 구축합니다.
    ```bash
    git worktree add ../tcn-feature-branch feat/feature-name
    ```

---

## 2. 브랜치 격리 및 메인 동기화 규칙
* **독립 기능 브랜치 사용:**
  * 모든 신규 개발 및 버그 수정은 반드시 개별 브랜치(`feat/기능명`, `fix/버그명`)를 먼저 분기하고 커밋해야 합니다. `main` 브랜치에 직접 커밋하는 것을 엄격히 금지합니다.
* **최신 main 동기화 (Safe Sync):**
  * `main` 브랜치의 최신 코드를 작업 브랜치로 가져와야 할 때, 작업 디렉토리의 변경사항이 유실되거나 충돌하는 것을 방지하기 위해 다음 순서를 준수합니다.
    ```bash
    git stash -u                  # 1. 작업 중인 모든 변경사항(untracked 포함)을 안전하게 대피
    git fetch origin              # 2. 원격 최신 로그 가져오기
    git merge origin/main         # 3. 내 브랜치에 main 병합
    git stash pop                 # 4. 대피시켰던 작업 파일들을 다시 복원 및 병합
    ```

## 3. 동시 편집 파일의 조율
* **전역 공통 파일 제어:**
  * [src/i18n/ui.ts](file:///Users/jhkim/Desktop/work/tcn/src/i18n/ui.ts) 또는 `package.json` 같이 여러 기능 개발 과정에서 공통으로 수정이 잦은 파일들은 작업 간 병목과 충돌의 주원인입니다.
* **해결책:**
  * 의존성 패키지 추가(`npm i`) 등의 작업은 가능하면 메인 드라이버 역할을 하는 에이전트가 단독으로 수행하여 `package-lock.json` 충돌을 원천 차단합니다.
  * 공통 다국어 키 추가 작업 시, PR 머지 직전에 병합 충돌 여부를 로컬에서 항상 먼저 확인하고 해결한 뒤 원격지에 올립니다.

## 4. PR 기반 품질 검증 및 머지 정책
* **직접 머지 금지:**
  * 원격지의 `main` 브랜치로 코드를 집어넣을 때는 로컬에서 직접 머지하여 push하지 않고, 반드시 **GitHub Pull Request (PR)**를 생성합니다.
* **CI 검증 게이트 통과:**
  * PR이 생성되면 다음 자동 검증 파이프라인([.github/workflows/ci.yml](file:///Users/jhkim/Desktop/work/tcn/.github/workflows/ci.yml))이 정상적으로 성공(Green)하는지 확인해야 합니다.
    * `npm run test` (유닛 테스트 및 커버리지 검증)
    * `npm run check` (Astro/TS 문법 및 타입 체크)
    * `npm run i18n` (한·영 번역 정합성 및 누락 검사)
* **알림 수신 확인:**
  * CI 결과에 따라 디스코드 및 텔레그램 채널로 전송되는 빌드 성공/실패 여부를 수신한 뒤 최종 머지를 실행합니다.
